# Support

Use this page to figure out where to ask for help and where to report problems.

If you need the current public support boundary or maintainer ownership, read [MAINTAINERS.md](MAINTAINERS.md) first.

## Before you open an issue

Start with the docs that match your setup:

- [README.md](README.md)
- [QUICKSTART.md](QUICKSTART.md)
- [docs/urgentry-tiny-build-guide.md](docs/urgentry-tiny-build-guide.md)
- [docs/urgentry-tiny-deploy-guide.md](docs/urgentry-tiny-deploy-guide.md)
- [docs/urgentry-operations-guide.md](docs/urgentry-operations-guide.md)

If you are working on the serious self-hosted preview, read:

- [docs/urgentry-serious-self-hosted-deployment-guide.md](docs/urgentry-serious-self-hosted-deployment-guide.md)

## Which issue should I open?

- Product bug, runtime bug, packaging problem: use the bug report template.
- Missing or wrong documentation: use the docs template.
- Tiny-mode evaluation friction or analytics UX confusion: use the design-partner feedback template.
- Product or workflow idea: use the feature request template.

When you file an issue, include:

- Tiny mode or serious self-hosted preview
- how you installed or ran Urgentry
- the version or commit you tested
- exact steps to reproduce the problem

## Security issues

Do not file security issues in public.

Use the private reporting path in [SECURITY.md](SECURITY.md).

## What this repo does not promise yet

Tiny mode is the public default. Serious self-hosted is still the preview path.

If you report a serious self-hosted problem, say so up front. That changes the expected setup, logs, and deployment details.

If you are evaluating Tiny mode with us directly, start with [docs/urgentry-tiny-feedback-loop.md](docs/urgentry-tiny-feedback-loop.md). It explains which reports should land as bugs, docs issues, or design-partner feedback.
